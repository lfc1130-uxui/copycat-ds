# Contributing to Copycat Design System

## File ownership map

Understanding which file **owns** data vs. which **copies** it is the key to keeping things in sync. When a source changes, every consumer must follow.

```
SOURCE OF TRUTH                 CONSUMERS (must be kept in sync)
───────────────                 ────────────────────────────────
src/styles/tokens.css     →     index.html (inline <style>)
                          →     figma-plugin/ui.html (inline <style>)

components/button.json    →     figma-plugin/ui.html (EXAMPLES.button)
components/badge.json     →     figma-plugin/ui.html (EXAMPLES.badge)
components/icon.json      →     figma-plugin/ui.html (EXAMPLES.icon)   ← generated by sync script

src/components/Icon/
  icons.ts                →     components/icon.json                    ← sync: icon-json
                          →     index.html (ICON_PATHS object)          ← sync: icon-paths

src/components/Button/
  Button.tsx + Button.css →     index.html (SCHEMAS.button, demo renderer)
                          →     playground/src/App.tsx (usage examples)

src/components/Badge/
  Badge.tsx + Badge.css   →     index.html (SCHEMAS.badge, demo renderer)
                          →     playground/src/App.tsx (usage examples)
```

The two hardest files to keep in sync are `index.html` and `figma-plugin/ui.html` because they embed source data inline — they have no build step that pulls from the canonical files.

## Change checklists

### Renaming or adding a design token

1. `src/styles/tokens.css` — add/rename in `:root` and `.dark`
2. `index.html` — find and update the inline `<style>` block (search for the old token name)
3. `figma-plugin/ui.html` — update any inline styles and the `EXAMPLES.*` token name strings
4. `components/button.json` or `components/badge.json` — update `"text"`, `"bg"`, `"border"` token fields
5. Any `.css` files under `src/components/` that reference the token

**Token naming convention:** `color/{role}/{modifier}` — e.g. `color/text/primary`, `color/bg/disabled`, `color/brand/hover`. Never use `color/fg/*` — that prefix is retired.

---

### Adding a new component

1. Create `src/components/{Name}/{Name}.tsx`, `{Name}.css`, `index.ts`
2. Export from `src/index.ts`
3. Create `components/{name}.json` — component spec consumed by the Figma plugin
4. Add `EXAMPLES.{name}` entry in `figma-plugin/ui.html`
5. Add an `<button class="btn-example" data-example="{name}">` in the plugin UI
6. Add a `#view-{name}` section to `index.html` (nav item + view + props table + usage block)
7. Add demo rows to `playground/src/App.tsx`
8. Create `src/components/{Name}/{Name}.figma.tsx` for Code Connect when the Figma component exists

---

### Adding or changing a component variant/prop

1. Update `src/components/{Name}/{Name}.tsx` — add prop, update types
2. Update `src/components/{Name}/{Name}.css` — add variant CSS
3. Update `components/{name}.json` — add to `axes`, `sizes`, or `styles`
4. Update `figma-plugin/ui.html` — add state to `EXAMPLES.{name}.axes` and `styles`
5. Update `index.html` — add to `SCHEMAS.{name}.axes`, update `makeEl()` renderer, add prop row to the props table, add usage example
6. Update `playground/src/App.tsx` — add demo row

---

### Adding icons

1. `src/components/Icon/icons.ts` — add entry to the `icons` object (kebab-case name, 24×24 Lucide-style path)
2. Run `npm run sync` — this regenerates `components/icon.json` and patches `figma-plugin/ui.html` and `index.html` ICON_PATHS automatically
3. `index.html` — manually add the icon name to the `ICON_GROUPS` array in the correct category, and update the sidebar badge count
4. `playground/src/App.tsx` — add the name to the appropriate `ICON_GROUPS` array

`components/icon.json` is the canonical Figma schema. Load it in the plugin via **Components → Load example → Icon → Generate** to regenerate the Figma component set. Icon names must match exactly — the Figma plugin creates variants named `Name={kebab-name}`, and `getIconSet()` in `code.js` looks them up by that key.

---

### Changing disabled state behavior

This has burned us before (opacity vs. explicit color). Changes touch:

1. `src/components/{Name}/{Name}.css` — the `:disabled` CSS rules
2. `src/styles/tokens.css` — disabled tokens (`color/bg/disabled`, `color/text/disabled`, `color/border/disabled`)
3. `components/{name}.json` — `styles.*.Disabled` hex values and token names
4. `index.html` — inline CSS disabled rules + demo stage
5. `figma-plugin/ui.html` — `EXAMPLES.{name}.styles.*.Disabled` values

## Sync strategy (current)

There is no automated sync script yet. Until one exists, the process is **manual with this document as the checklist**.

When editing `index.html` or `figma-plugin/ui.html`, search for the comment marker closest to what you're changing:

```
index.html markers:
  /* sync-start:tokens */  /* sync-end:tokens */
  /* sync-start:button-css */  /* sync-end:button-css */
  /* sync-start:badge-css */  /* sync-end:badge-css */
  // sync-start:icon-paths  // sync-end:icon-paths

ui.html markers:
  // sync-start:icon-examples  // sync-end:icon-examples
```

Run `npm run sync` to patch all five. Run `npm run sync:dry` to preview without writing. Pass a target name to patch only one: `npm run sync tokens`.

Future plan: a `scripts/sync-docs.js` that reads the source files and patches `index.html` and `ui.html` between these markers automatically.

## Playground

```bash
cd playground
npm install
npm run dev
```

The playground at `playground/src/App.tsx` is the fastest way to visually verify component changes. Always add a row for any new variant or state you introduce.

## Figma plugin

Load the plugin from `figma-plugin/` in Figma's developer tools. The plugin has two modes:

- **Variables** — imports/exports design tokens from `tokens/tokens.json`
- **Components** — generates Figma component sets from `components/*.json` schemas; use the "Load example" buttons to populate the textarea, then click Generate

When `code.js` or `ui.html` changes, reload the plugin in Figma (there is no hot reload).
